# Concatenation Woes: Search Dereferencing and WOCONSTANIC Race Conditions

> AI agents: read [../DOC_AGENTS.md](../DOC_AGENTS.md) before editing this file.

**Status**: Design Discussion → Resolution
Date: 2026-02-20
**Context**: Identified during review of UBC2 design specification

## Summary

This document captures the design discussion that led to the **search dereferencing microstate protocol** for UBC2. The key insights:

1. **Identifiers are searches** - RHS identifiers are SearchFir instances that perform search operations, not dereferenced variables
2. **Dereferencing loops chase through WOCONSTANIC, stop at CONSTANIC** - WOCONSTANIC searches are transparent (dereference through), CONSTANIC searches are barriers (stop and wait)
3. **SearchFir is driven by proto-brane Nyes states** - SearchFir has a simple `resolve()` method called by proto-brane during EMBRYONIC; proto-brane orchestrates the lifecycle
4. **Constanic cloning enables re-coordination** - CONSTANIC clones → EMBRYONIC (re-search), WOCONSTANIC clones → BRANING (wait for children, then re-dereference)
5. **Parent chain updates during cloning** - Clones get new parent references so searches resolve in the new context

These designs have been integrated into the main UBC2 design specification.

## Foundational Clarification: Identifiers Are Searches

**IMPORTANT**: An identifier on the RHS of an assignment is **always a search operation**. There
is no "dereferencing" of identifiers in the traditional sense. When you write:

```foolish
a = x;
```

The `x` is not "looked up and dereferenced" — it is a **SearchFir** that performs a search
operation and stores the result.

**We never dereference identifiers.** Identifiers ARE searches. What we're discussing here is:
- What does a search FIR store when it finds something?
- What state does the search FIR transition to based on what it found?
- How does the search FIR's `value()` method work?

All identifiers on RHS are SearchFir instances. All resolution is search resolution.

## The Core Issues

### Issue 1: Search Dereferencing Protocol Missing

The UBC2 design does not specify how search results are dereferenced when they point to other
searches. This creates ambiguity about what happens in chains like:

```foolish
{
    a = x;      // x is CONSTANIC (not found)
    b = a;      // search for 'a' finds statement 0
    c = b;      // search for 'b' finds statement 1
    d = c;      // search for 'c' finds statement 2
    e = d;      // search for 'd' finds statement 3
    f = e;      // search for 'e' finds statement 4
}
```

**Question**: What state does each statement reach?

### Issue 2: WOCONSTANIC Race Condition

Consider:

```foolish
{
    a = x1 + x2 + x3 + x4 + x5;
    b = x1 + y1 + y2 + y3;
    c = a + b;
}
```

The brane's statement `c = a + b` can become **WOCONSTANIC before a and b become CONSTANIC**.

**Why?** Because:
1. During EMBRYONIC, the search for `a` **finds** identifier 'a' at statement 0
2. The search for `b` **finds** identifier 'b' at statement 1
3. Both searches succeed immediately (the identifiers exist)
4. Statement c transitions to WOCONSTANIC (found everything, waiting on dependencies)
5. Meanwhile, statements a and b are still in EMBRYONIC, searching for x1, x2, x3, x4, x5, y1, y2, y3
6. a and b may take many steps to determine all the x's and y's are not found
7. Eventually a and b transition to CONSTANIC (searches failed)

**Current UBC2 design does not address this race condition.**

## Cross-Reference Table: What Happens When a Search Finds Something

When a SearchFir successfully finds an identifier, it retrieves a FIR from the parent brane's
statement array. That FIR can be one of three types:

1. **Unanchored Search** (another SearchFir performing unanchored search: `x`, `pattern`)
2. **Anchored Search** (SearchFir with anchor: `brane?member`, `{#-1}`, `brane$`)
3. **Brane** (proto-brane or full brane)

And each can be in one of these states:

- **PREMBRYONIC** (just instantiated, hasn't taken the atomic step yet)
- **EMBRYONIC** (actively searching and communicating)
- **BRANING** (stepping child branes)
- **CONSTANIC** (search failed, nothing found)
- **WOCONSTANIC** (searches succeeded, waiting on constanic dependencies)
- **CONSTANT** (fully resolved)
- **INDEPENDENT** (literal value or promoted CONSTANT)

### Dereferencing Loop Approach (User's Proposal)

The key insight: **searches don't just check the immediate target state — they chase through search
chains until they find the "real" result.**

Here's the algorithm in pseudocode:

```python
def resolve_search(search_fir):
    cur_res = None

    while True:
        # Step 1: If no result yet, perform the search
        if cur_res is None:
            cur_res = search_for_identifier(search_fir.pattern)
            if cur_res is None:
                # Search failed - nothing found
                search_fir.transition_to(CONSTANIC)
                return

        # Step 2: We found something. What is it?

        # Case A: Found a search
        if cur_res.is_search():
            # If the found search is still stepping, wait for it
            if cur_res.is_pre_constanic():  # PREMBRYONIC, EMBRYONIC, BRANING
                search_fir.add_to_wait_for_queue()
                return  # Check again later

            # If the found search is CONSTANIC (failed search)
            if cur_res.at_constanic():
                search_fir.transition_to(CONSTANIC)
                return

            # If the found search is WOCONSTANIC (found something, waiting)
            if cur_res.at_woconstanic():
                # Chase the chain: dereference to what it found
                cur_res = cur_res.get_search_result()
                continue  # Loop again with the dereferenced result

            # If the found search is CONSTANT
            if cur_res.at_constant():
                # Dereference to final value
                search_fir.store_reference(cur_res)
                search_fir.transition_to(CONSTANT)
                return

        # Case B: Found a brane (proto or full)
        else:  # cur_res.is_brane()
            # If the brane is still stepping, wait for it
            if cur_res.is_pre_constanic():
                search_fir.add_to_wait_for_queue()
                return

            # If the brane is CONSTANIC
            if cur_res.at_constanic():
                # We found the brane, but it's unresolved
                # Store reference and become WOCONSTANIC (found it, waiting on it)
                search_fir.store_reference(cur_res)
                search_fir.transition_to(WOCONSTANIC)
                return

            # If the brane is WOCONSTANIC
            if cur_res.at_woconstanic():
                # We found the brane, it's waiting on dependencies
                search_fir.store_reference(cur_res)
                search_fir.transition_to(WOCONSTANIC)
                return

            # If the brane is CONSTANT or INDEPENDENT
            if cur_res.at_constant() or cur_res.at_independent():
                # Fully resolved brane
                search_fir.store_reference(cur_res)
                search_fir.transition_to(CONSTANT)
                return
```

### Key Differences from Claude's Initial Suggestion

1. **Searches are chased through WOCONSTANIC chains**: When a search finds another search that's
   WOCONSTANIC, it **dereferences** to what that search found and continues the loop. This creates
   a dereferencing chain.

2. **Searches that find CONSTANIC searches become CONSTANIC**: If the chain ends at a CONSTANIC
   search (failed search), the original search also becomes CONSTANIC (not WOCONSTANIC). The
   "unfound" state propagates backward through the chain.

3. **Branes don't dereference through WOCONSTANIC**: When a search finds a WOCONSTANIC brane, it
   stops and becomes WOCONSTANIC itself. Branes are not "transparent" like searches — they're
   endpoints.

### Example: Search Chain

```foolish
{
    a = x;      // x is CONSTANIC (not found)
    b = a;      // search for 'a'
    c = b;      // search for 'b'
}
```

**Step-by-step for `c = b`:**

1. `cur_res = None`
2. Search for `b` → finds SearchFir(a) at statement 1
3. `cur_res = SearchFir(a)`, it's a search
4. `SearchFir(a).at_woconstanic()` → Yes (it found 'a', which is CONSTANIC)
5. `cur_res = SearchFir(a).get_search_result()` → returns SearchFir(x)
6. Loop again: `cur_res = SearchFir(x)`, it's a search
7. `SearchFir(x).at_constanic()` → Yes (search failed)
8. `SearchFir(c).transition_to(CONSTANIC)`

**Result**: `c` becomes CONSTANIC (not WOCONSTANIC), because the chain terminates at a failed search.

### Updated Table (Based on Dereferencing Loop)

| Found FIR Type | PREMBRYONIC | EMBRYONIC | BRANING | CONSTANIC | WOCONSTANIC | CONSTANT | INDEPENDENT |
|----------------|---------------|-----------|---------|-----------|-------------|----------|-------------|
| **Unanchored Search** | nigh (wait-for) | nigh (wait-for) | nigh (wait-for) | **CONSTANIC** (chase → failed) | **Continue loop** (dereference) | CONSTANT (reference) | CONSTANT (reference) |
| **Anchored Search** | nigh (wait-for) | nigh (wait-for) | nigh (wait-for) | **CONSTANIC** (chase → failed) | **Continue loop** (dereference) | CONSTANT (reference) | CONSTANT (reference) |
| **Brane (proto)** | nigh (wait-for) | nigh (wait-for) | nigh (wait-for) | WOCONSTANIC (found, unresolved) | WOCONSTANIC (found, waiting) | CONSTANT (reference) | CONSTANT (reference) |
| **Brane (full)** | nigh (wait-for) | nigh (wait-for) | nigh (wait-for) | WOCONSTANIC (found, unresolved) | WOCONSTANIC (found, waiting) | CONSTANT (reference) | CONSTANT (reference) |

**Key insight**: Searches are **transparent** — WOCONSTANIC searches are dereferenced through.
Branes are **opaque** — WOCONSTANIC branes stop the dereferencing loop.

## Architectural Decision: Where Does the Dereferencing Loop Live?

### The Problem

Consider:
```foolish
{
    a = (b + c + d);
}
```

The expression `b + c + d` desugars to `{b}{c}{d}{🧠+}{🧠+}` which concatenates to a proto-brane
with statements `{b, c, d, 🧠+, 🧠+}`.

Each of `b`, `c`, `d` is a SearchFir. When the proto-brane steps through EMBRYONIC, these searches
are performed. But **who** does the dereferencing loop?

### Option A: SearchFir Is Smart (Has Its Own State Machine)

In this approach, SearchFir is a full-fledged FIR with its own state machine. When a search finds
a result:

```
SearchFir performs search → finds target
SearchFir checks target state
If target is WOCONSTANIC search → SearchFir dereferences and continues loop
If target is CONSTANIC search → SearchFir becomes CONSTANIC
If target is WOCONSTANIC brane → SearchFir becomes WOCONSTANIC
...etc
```

**Pros:**
- SearchFir encapsulates all search logic
- Each search manages its own dereferencing
- Proto-brane just says "step all my searches" and they handle themselves

**Cons:**
- SearchFir becomes complex with its own lifecycle
- Dereferencing logic duplicated across all search types (unanchored, anchored, etc.)
- SearchFir has to track state (nigh → EMBRYONIC → searching → dereferencing → constanic)
- More FIR types with independent state machines = harder to reason about

### Option B: SearchFir Is Dumb (Just a Search Holder)

In this approach, SearchFir is a **simple data structure** that holds:
- The search pattern (identifier name, regex, etc.)
- Whether the search is anchored or unanchored
- The found result (once resolved)

The **proto-brane** (or the parent brane) performs the dereferencing loop during EMBRYONIC:

```
Proto-brane EMBRYONIC step:
    for each SearchFir in my expressions:
        result = perform_search(SearchFir.pattern)
        if result is None:
            SearchFir.state = CONSTANIC
            continue

        # Dereferencing loop (proto-brane handles this)
        while result.is_search() and result.at_woconstanic():
            result = result.get_search_result()

        # After dereferencing, check final result state
        if result.is_search() and result.at_constanic():
            SearchFir.state = CONSTANIC
        elif result.is_pre_constanic():
            add SearchFir to wait-for queue
        elif result.at_constanic() or result.at_woconstanic():
            SearchFir.state = WOCONSTANIC
            SearchFir.target = result
        elif result.at_constant() or result.at_independent():
            SearchFir.state = CONSTANT
            SearchFir.target = result
```

**Pros:**
- SearchFir is simple: just holds pattern and result
- Dereferencing logic centralized in proto-brane EMBRYONIC stage
- All searches (in binary ops, unary ops, plain identifiers) handled uniformly
- Proto-brane already orchestrates EMBRYONIC communication — this is one more thing it orchestrates

**Cons:**
- Proto-brane becomes more complex (handles dereferencing for all children searches)
- SearchFir is "dumb" — doesn't know how to resolve itself
- Violates single-responsibility: proto-brane both manages children AND dereferences searches

### Option C: Hybrid (SearchFir Has Helper, Proto-Brane Drives)

SearchFir has a `resolve()` method that performs the dereferencing loop, but **proto-brane calls
it** during EMBRYONIC:

```java
class SearchFir {
    Pattern pattern;
    FIR target;
    Nyes state;

    // Called by proto-brane during EMBRYONIC
    void resolve(ParentBrane parent) {
        FIR cur_res = parent.search(this.pattern);
        if (cur_res == null) {
            this.state = CONSTANIC;
            return;
        }

        // Dereferencing loop
        while (cur_res.is_search() && cur_res.at_woconstanic()) {
            cur_res = cur_res.get_search_result();
        }

        // Determine state based on final result
        if (cur_res.is_search() && cur_res.at_constanic()) {
            this.state = CONSTANIC;
        } else if (cur_res.is_pre_constanic()) {
            parent.add_to_wait_for_queue(this);
        } else if (cur_res.at_constanic() || cur_res.at_woconstanic()) {
            this.state = WOCONSTANIC;
            this.target = cur_res;
        } else if (cur_res.at_constant() || cur_res.at_independent()) {
            this.state = CONSTANT;
            this.target = cur_res;
        }
    }
}
```

Proto-brane EMBRYONIC:
```java
for (SearchFir search : mySearches) {
    search.resolve(this);
}
```

**Pros:**
- SearchFir encapsulates dereferencing logic
- Proto-brane just drives the process (calls resolve on all searches)
- Clear separation: SearchFir knows **how** to resolve; proto-brane knows **when** to resolve
- SearchFir doesn't need its own step() lifecycle — proto-brane steps it explicitly

**Cons:**
- SearchFir is not fully autonomous (requires proto-brane to call resolve)
- Slightly more complex than fully dumb SearchFir

## User's Concern: Parentheses `()` and Proto-Branes

User asks: Is `(b+c+d)` a special brane?

**Answer**: No! Parentheses `()` are just **grouping for precedence** in the parser. They affect
how the AST is built, but once we have the AST, there's no "parenthesis brane."

The expression `b+c+d` becomes:
```
BinaryOp(
    BinaryOp(SearchFir(b), SearchFir(c), 🧠+),
    SearchFir(d),
    🧠+
)
```

Or in the desugared view: `{b, c, 🧠+, d, 🧠+}` (a single proto-brane via concatenation).

When this proto-brane reaches EMBRYONIC, it has to resolve `b`, `c`, and `d`. The proto-brane
(or something) must perform the dereferencing loop for each search.

## Recommendation: Option B (Dumb SearchFir) or Option C (Hybrid)

**Lean toward Option C (Hybrid)** for these reasons:

1. **Matches UBC2's philosophy**: Proto-brane orchestrates EMBRYONIC stage. It makes sense for
   proto-brane to drive search resolution by calling `search.resolve(this)`.

2. **SearchFir doesn't need independent lifecycle**: It's not a proto-brane itself. It's a
   component of a proto-brane. The proto-brane's lifecycle is sufficient.

3. **Centralized dereferencing logic**: The dereferencing loop lives in one place
   (`SearchFir.resolve()`), not scattered across different search types.

4. **Simple interface**: Proto-brane just calls `resolve()` on all its searches during EMBRYONIC.
   SearchFir handles the details.

### Concrete Implementation Sketch (Option C)

**SearchFir.java:**
```java
class SearchFir extends FIR {
    Query query;           // Pattern, anchored/unanchored, etc.
    FIR target;            // The found result (after dereferencing)
    Nyes state;            // CONSTANIC, WOCONSTANIC, CONSTANT

    // Called by parent proto-brane during EMBRYONIC
    void resolve(ProtoBrane parent) {
        FIR cur_res = parent.performSearch(this.query);

        if (cur_res == null) {
            // Search failed - nothing found
            this.state = CONSTANIC;
            return;
        }

        // Dereferencing loop: chase through WOCONSTANIC searches
        int depth = 0;
        while (cur_res instanceof SearchFir && cur_res.at_woconstanic()) {
            cur_res = ((SearchFir) cur_res).target;
            depth++;
            if (depth > MAX_DEREFERENCE_DEPTH) {
                throw new CircularReferenceException("Search chain too deep");
            }
        }

        // Determine state based on final dereferenced result
        if (cur_res instanceof SearchFir && cur_res.at_constanic()) {
            // Chain ends at failed search
            this.state = CONSTANIC;
        } else if (cur_res.is_pre_constanic()) {
            // Target still stepping - wait for it
            parent.addToWaitForQueue(this);
            this.state = pre-constanic;  // remains nigh until target resolves  // Still waiting
        } else if (cur_res.at_constanic() || cur_res.at_woconstanic()) {
            // Found a brane (or non-search) that's constanic
            this.state = WOCONSTANIC;
            this.target = cur_res;
        } else if (cur_res.at_constant() || cur_res.at_independent()) {
            // Fully resolved
            this.state = CONSTANT;
            this.target = cur_res;
        }
    }

    @Override
    public Object value() {
        if (at_constant()) return target.value();
        if (at_woconstanic()) return target;  // Return the FIR we're waiting on
        if (at_constanic()) return this;      // Return self as CONSTANIC marker
        throw new IllegalStateException("Cannot get value of nigh SearchFir");
    }
}
```

**ProtoBrane EMBRYONIC step:**
```java
void stepEmbryonic() {
    // Resolve all searches (up to search bandwidth)
    int searchesProcessed = 0;
    for (SearchFir search : pendingSearches) {
        if (searchesProcessed >= SEARCH_BANDWIDTH) break;

        search.resolve(this);
        searchesProcessed++;
    }

    // Handle communication, wait-for queue, etc.
    // ...
}
```

### Answers to User's Questions

1. **Depth limit**: Yes, include `MAX_DEREFERENCE_DEPTH` to prevent infinite loops (e.g., 100).

2. **get_search_result()**: Rename to just `target` field. Only valid when state is WOCONSTANIC or
   CONSTANT.

3. **Anchored search into CONSTANIC brane**: The brane is CONSTANIC, meaning it has unresolved
   searches. But we **can** still search inside it! The brane's statement array exists. The
   anchored search `b.x` would look for `x` in brane `b`'s statements. If `x` exists, the search
   succeeds. This is important for partial evaluation.

4. **Proto-brane vs full brane**: The dereferencing loop doesn't distinguish. It only cares about
   FIR state. The `value()` method handles proto-brane (returns scalar) vs full brane (returns
   brane).

## Critical Clarification: When Does Dereferencing Stop?

### The Key Insight

**CONSTANIC is a barrier for dereferencing.** When the dereferencing loop encounters a CONSTANIC
search, it **stops** and the original search becomes WOCONSTANIC (waiting for that CONSTANIC to
resolve through re-coordination).

**WOCONSTANIC searches should be dereferenced through.** They found something, and we should keep
chasing to see what they found.

### Updated Dereferencing Rules

A search goes through these phases:

1. **EMBRYONIC/BRANING (active searching)**: Performing search and dereferencing
2. **Finished dereferencing** → transitions to terminal state:
   - **CONSTANIC**: Search found nothing, OR dereferencing chain ended at CONSTANIC search
   - **WOCONSTANIC**: Dereferencing chain ended at CONSTANIC or WOCONSTANIC non-search (brane)
   - **CONSTANT**: Dereferencing chain ended at CONSTANT or INDEPENDENT result

### The Corrected Dereferencing Loop

```python
def resolve_search(search_fir):
    # Perform initial search
    cur_res = search_for_identifier(search_fir.pattern)

    if cur_res is None:
        # Search failed - nothing found
        search_fir.transition_to(CONSTANIC)
        return

    # Dereferencing loop: chase through searches UNTIL we hit CONSTANIC
    depth = 0
    while cur_res.is_search():
        depth += 1
        if depth > MAX_DEREFERENCE_DEPTH:
            raise CircularReferenceException()

        # Check state of the found search
        if cur_res.at_constanic():
            # STOP: CONSTANIC is a barrier
            # The original search becomes WOCONSTANIC (waiting on this CONSTANIC to resolve)
            search_fir.transition_to(WOCONSTANIC)
            search_fir.target = cur_res
            return

        if cur_res.at_woconstanic():
            # CONTINUE: Dereference through WOCONSTANIC searches
            # Get what this WOCONSTANIC search is pointing to
            cur_res = cur_res.target
            continue

        if cur_res.at_constant():
            # STOP: Found fully resolved search
            search_fir.transition_to(CONSTANT)
            search_fir.target = cur_res.target  # Dereference to final value
            return

        if cur_res.is_pre_constanic():  # nigh (EMBRYONIC, BRANING, etc.)
            # STOP: Target still stepping, wait for it
            search_fir.add_to_wait_for_queue()
            search_fir.state = pre-constanic  # remains nigh
            return

    # If we exit the loop, cur_res is NOT a search (it's a brane)
    # Check the brane's state
    if cur_res.at_constanic() or cur_res.at_woconstanic():
        # Found a brane that's constanic - we're waiting on it
        search_fir.transition_to(WOCONSTANIC)
        search_fir.target = cur_res
        return

    if cur_res.at_constant() or cur_res.at_independent():
        # Found a fully resolved brane
        search_fir.transition_to(CONSTANT)
        search_fir.target = cur_res
        return

    if cur_res.is_pre_constanic():
        # Brane still stepping
        search_fir.add_to_wait_for_queue()
        search_fir.state = pre-constanic  # remains nigh
        return
```

### Key Changes from Previous Version

1. **CONSTANIC search stops dereferencing**: When we encounter `cur_res.at_constanic()`, we STOP
   and the original search becomes WOCONSTANIC. Why? Because CONSTANIC means "search failed and
   will never succeed unless re-coordinated." There's no point dereferencing further.

2. **WOCONSTANIC search continues dereferencing**: When we encounter `cur_res.at_woconstanic()`,
   we dereference to `cur_res.target` and continue the loop. Why? Because WOCONSTANIC means "I
   found something, just waiting for it to resolve." We should chase to see what it found.

3. **Original search never becomes CONSTANIC from dereferencing a CONSTANIC search**: If the
   chain ends at a CONSTANIC search, the original search becomes **WOCONSTANIC** (not CONSTANIC).
   Why? Because the original search DID find something (the identifier exists). It's just waiting
   for that thing to resolve.

### Example: Search Chain Ending at CONSTANIC

```foolish
{
    a = x;      // x not found → SearchFir(a) is CONSTANIC
    b = a;      // search for 'a'
    c = b;      // search for 'b'
}
```

**SearchFir(c).resolve():**

1. Search for 'c' → not found... wait, search for 'b' → finds SearchFir(b) at statement 1
2. `cur_res = SearchFir(b)`, it's a search, check state...
3. SearchFir(b) is nigh (hasn't resolved yet) → SearchFir(c) waits

**SearchFir(b).resolve():**

1. Search for 'b'... wait, search for 'a' → finds SearchFir(a) at statement 0
2. `cur_res = SearchFir(a)`, it's a search, check state...
3. SearchFir(a) is nigh → SearchFir(b) waits

**SearchFir(a).resolve():**

1. Search for 'a'... wait, search for 'x' → not found
2. SearchFir(a).state = CONSTANIC

**Next EMBRYONIC step - re-check wait-for queue:**

**SearchFir(b).resolve() (second time):**

1. Search for 'a' → finds SearchFir(a) at statement 0
2. `cur_res = SearchFir(a)`, it's a search, check state...
3. SearchFir(a).at_constanic() → **TRUE**
4. **STOP dereferencing** - CONSTANIC is a barrier
5. SearchFir(b).state = **WOCONSTANIC** (waiting on CONSTANIC SearchFir(a))
6. SearchFir(b).target = SearchFir(a)

**SearchFir(c).resolve() (second time):**

1. Search for 'b' → finds SearchFir(b) at statement 1
2. `cur_res = SearchFir(b)`, it's a search, check state...
3. SearchFir(b).at_woconstanic() → **TRUE**
4. **CONTINUE dereferencing** - dereference to what SearchFir(b) is waiting on
5. `cur_res = SearchFir(b).target` → SearchFir(a)
6. `cur_res = SearchFir(a)`, it's a search, check state...
7. SearchFir(a).at_constanic() → **TRUE**
8. **STOP dereferencing** - CONSTANIC is a barrier
9. SearchFir(c).state = **WOCONSTANIC** (waiting on CONSTANIC SearchFir(a))
10. SearchFir(c).target = SearchFir(a)

**Result:**
- `a` is **CONSTANIC** (search for 'x' failed)
- `b` is **WOCONSTANIC** (found 'a', but 'a' is CONSTANIC)
- `c` is **WOCONSTANIC** (found 'b', dereferenced to 'a', 'a' is CONSTANIC)

**Rendering:**
```
{
＿a = ?;
＿b = ??;
＿c = ??;
}
```

### Why This Matters: CONSTANIC Will Never Change Without Re-Coordination

CONSTANIC means "I searched and found nothing in my current context." This state **will not
change** unless the FIR is re-coordinated into a new context (via constanic cloning).

So when dereferencing encounters a CONSTANIC search, there's no point waiting for it to "become
CONSTANT" — that will never happen in the current context. The search should stop dereferencing
and become WOCONSTANIC (waiting for re-coordination to provide a new context).

### Updated Summary: Search Terminal States

A search can end in one of three terminal states:

1. **CONSTANIC**: The search itself found nothing (identifier doesn't exist)
2. **WOCONSTANIC**: The dereferencing chain ended at a CONSTANIC or WOCONSTANIC FIR
3. **CONSTANT**: The dereferencing chain ended at a CONSTANT or INDEPENDENT FIR

## Critical Question: Who Handles Re-Dereferencing When WOCONSTANIC Resolves?

### The Scenario

```foolish
{
    a = x;      // x is CONSTANIC (not found)
    b = a;      // search for 'a'
    c = b;      // search for 'b'
}
```

**Timeline:**

**EMBRYONIC Step 1:**
- SearchFir(c) calls `resolve()`:
  - Finds `b` at statement 1 → `cur_res = SearchFir(b)`
  - SearchFir(b) is **nigh** (hasn't resolved yet)
  - SearchFir(c) adds itself to wait-for queue
  - SearchFir(c).state = nigh

**EMBRYONIC Step 2:**
- SearchFir(b) calls `resolve()`:
  - Finds `a` at statement 0 → `cur_res = SearchFir(a)`
  - SearchFir(a) is **nigh** (hasn't resolved yet)
  - SearchFir(b) adds itself to wait-for queue
  - SearchFir(b).state = nigh

**EMBRYONIC Step 3:**
- SearchFir(a) calls `resolve()`:
  - Searches for `x` → not found
  - SearchFir(a).state = CONSTANIC

**Now what? SearchFir(b) and SearchFir(c) are in the wait-for queue. Who re-checks them?**

### Option C.1: Proto-Brane Handles Wait-For Queue Re-Checking

The proto-brane has a **wait-for queue** (as described in current UBC2 design). Every EMBRYONIC
step, the proto-brane checks the wait-for queue:

```java
void stepEmbryonic() {
    // Step 1: Resolve new searches
    for (SearchFir search : pendingSearches) {
        if (!search.hasBeenResolved()) {
            search.resolve(this);
        }
    }

    // Step 2: Re-check wait-for queue
    for (SearchFir search : waitForQueue) {
        // Try to resolve again - target might have changed state
        search.resolve(this);

        if (search.at_constanic() || search.at_woconstanic() || search.at_constant()) {
            // Search finalized - remove from wait-for queue
            waitForQueue.remove(search);
        }
    }

    // Step 3: Check if all searches resolved
    if (pendingSearches.isEmpty() && waitForQueue.isEmpty()) {
        transition(BRANING);
    }
}
```

**Key insight**: `search.resolve(this)` is **idempotent** (can be called multiple times). Each
time it's called, it re-performs the search and re-dereferences. If the target has changed state
(e.g., nigh → CONSTANIC), the search will detect this and update its own state.

**Example continuation:**

**EMBRYONIC Step 4:**
- Proto-brane checks wait-for queue
- SearchFir(b) calls `resolve()` again:
  - Finds `a` at statement 0 → `cur_res = SearchFir(a)`
  - SearchFir(a).at_constanic() → **True** (changed from nigh!)
  - Dereferencing loop: SearchFir(a) is a search and CONSTANIC → done
  - SearchFir(b).state = CONSTANIC (propagate)
  - SearchFir(b) removed from wait-for queue

**EMBRYONIC Step 5:**
- Proto-brane checks wait-for queue
- SearchFir(c) calls `resolve()` again:
  - Finds `b` at statement 1 → `cur_res = SearchFir(b)`
  - SearchFir(b).at_constanic() → **True** (changed from nigh!)
  - Dereferencing loop: SearchFir(b) is a search and CONSTANIC → done
  - SearchFir(c).state = CONSTANIC (propagate)
  - SearchFir(c) removed from wait-for queue

**EMBRYONIC Step 6:**
- All searches resolved, wait-for queue empty
- Proto-brane transitions to BRANING

### Option C.2: SearchFir Subscribes to Target State Changes (Callback)

Instead of proto-brane polling the wait-for queue, SearchFir could **subscribe** to its target:

```java
void resolve(ProtoBrane parent) {
    FIR cur_res = parent.performSearch(this.query);

    if (cur_res == null) {
        this.state = CONSTANIC;
        return;
    }

    // Dereference...
    while (cur_res instanceof SearchFir && cur_res.at_woconstanic()) {
        cur_res = ((SearchFir) cur_res).target;
    }

    if (cur_res.is_pre_constanic()) {
        // Subscribe to target - when it changes state, re-resolve
        cur_res.addStateChangeListener(() -> {
            this.resolve(parent);  // Re-resolve when target changes
        });
        this.state = pre-constanic;  // remains nigh until target resolves
    }
    // ... rest of resolve logic
}
```

**Pros:**
- Efficient: only re-resolves when target actually changes
- No polling overhead

**Cons:**
- Complex: requires callback/observer pattern
- Circular dependency management (what if target subscribes to this search?)
- UBC2 design explicitly avoids callbacks in initial implementation

### Option C.3: Hybrid - Proto-Brane Checks, SearchFir Tracks Dependencies

Proto-brane maintains wait-for queue, but SearchFir tracks **what it's waiting for**:

```java
class SearchFir {
    FIR waitingOn;  // The FIR this search is waiting for

    void resolve(ProtoBrane parent) {
        // ... perform search, dereference

        if (cur_res.is_pre_constanic()) {
            this.waitingOn = cur_res;  // Track dependency
            parent.addToWaitForQueue(this);
        }
    }

    boolean isReadyToResolve() {
        return waitingOn == null || waitingOn.hasConstanicity();
    }
}
```

Proto-brane only re-checks searches that are ready:

```java
void stepEmbryonic() {
    // Re-check only searches whose dependencies resolved
    for (SearchFir search : waitForQueue) {
        if (search.isReadyToResolve()) {
            search.resolve(this);
            if (search.at_constanic() || search.at_woconstanic() || search.at_constant()) {
                waitForQueue.remove(search);
            }
        }
    }
}
```

**Pros:**
- More efficient than blind polling (Option C.1)
- Simpler than callbacks (Option C.2)

**Cons:**
- SearchFir needs to track dependency
- Still checks queue every step (but skips non-ready searches)

## Recommendation: Option C.1 (Proto-Brane Polling) for Initial Implementation

**Why?**

1. **Matches current UBC2 design**: UBC2 already specifies "wait-for mechanism uses busy-checking"
   (from EMBRYONIC section). Option C.1 is exactly this.

2. **Simple and correct**: Proto-brane calls `search.resolve(this)` every EMBRYONIC step for all
   searches in wait-for queue. SearchFir.resolve() is idempotent - it re-checks the target state
   and updates accordingly.

3. **No new complexity**: Doesn't require callbacks, dependency tracking, or observer pattern.

4. **Future optimization**: Can later optimize to Option C.3 (track dependencies) or even Option
   C.2 (callbacks) without changing the interface.

### Updated Implementation Sketch (Option C.1)

**ProtoBrane EMBRYONIC:**
```java
void stepEmbryonic() {
    // Resolve new searches (up to search bandwidth)
    int searchesProcessed = 0;
    for (SearchFir search : pendingSearches) {
        if (searchesProcessed >= SEARCH_BANDWIDTH) break;
        search.resolve(this);
        searchesProcessed++;
    }

    // Re-check wait-for queue (re-resolve searches waiting on nigh targets)
    Iterator<SearchFir> iter = waitForQueue.iterator();
    while (iter.hasNext()) {
        SearchFir search = iter.next();

        // Re-resolve - target might have changed state
        search.resolve(this);

        // If search finalized, remove from wait-for queue
        if (search.at_constanic() || search.at_woconstanic() || search.at_constant()) {
            iter.remove();
        }
    }

    // Check transition to BRANING
    if (allSearchesResolved() && waitForQueue.isEmpty()) {
        transition(BRANING);
    }
}
```

**SearchFir.resolve() (modified to be idempotent):**
```java
void resolve(ProtoBrane parent) {
    // If already finalized, don't re-resolve
    if (this.at_constanic() || this.at_woconstanic() || this.at_constant()) {
        return;
    }

    FIR cur_res = parent.performSearch(this.query);

    if (cur_res == null) {
        this.state = CONSTANIC;
        return;
    }

    // Dereferencing loop
    while (cur_res instanceof SearchFir && cur_res.at_woconstanic()) {
        cur_res = ((SearchFir) cur_res).target;
    }

    // Determine state
    if (cur_res instanceof SearchFir && cur_res.at_constanic()) {
        this.state = CONSTANIC;
    } else if (cur_res.is_pre_constanic()) {
        // Still waiting - stay in wait-for queue
        parent.addToWaitForQueue(this);
        this.state = pre-constanic;  // remains nigh until target resolves
    } else if (cur_res.at_constanic() || cur_res.at_woconstanic()) {
        this.state = WOCONSTANIC;
        this.target = cur_res;
    } else if (cur_res.at_constant() || cur_res.at_independent()) {
        this.state = CONSTANT;
        this.target = cur_res;
    }
}
```

**Key detail**: The first line of `resolve()` checks if already finalized. This prevents re-doing
work on searches that already reached terminal states.

## What About Extending Dereferencing When WOCONSTANIC Becomes CONSTANT?

Consider:
```foolish
{
    a = x1 + x2;  // Takes time to resolve
    b = a;        // Finds 'a', it's nigh, waits
    c = b;        // Finds 'b', it's nigh, waits
}
```

**Timeline:**

1. SearchFir(a) resolves x1, x2 → eventually becomes CONSTANT (value = 3)
2. SearchFir(b) re-resolves → finds SearchFir(a), it's CONSTANT → dereferences → becomes CONSTANT
3. SearchFir(c) re-resolves → finds SearchFir(b), it's CONSTANT → dereferences → becomes CONSTANT

**The dereferencing loop extends automatically** because `resolve()` is called again when targets
change state. No special handling needed!

**What About WOCONSTANIC Waiting on CONSTANIC?**

Here's the critical case:

```foolish
{
    a = x;     // x not found → SearchFir(a) is CONSTANIC
    b = a;     // Finds 'a', dereferences, hits CONSTANIC barrier → SearchFir(b) is WOCONSTANIC
}
```

SearchFir(b) is **WOCONSTANIC** waiting on SearchFir(a) which is **CONSTANIC**.

**Will SearchFir(a) ever transition from CONSTANIC to CONSTANT?**

**NO!** Not in the current context. CONSTANIC means "search failed in current context." The only
way for CONSTANIC to gain value is through **re-coordination** (constanic cloning into a new
context that provides the missing binding).

**So what happens to SearchFir(b)?**

SearchFir(b) stays **WOCONSTANIC** forever (in this context). It's waiting for SearchFir(a) to
resolve, but SearchFir(a) is CONSTANIC and won't change.

**This is correct!** SearchFir(b) DID find what it was looking for ('a' exists). It's just that
'a' itself is unresolved. When the entire brane reaches a terminal state, SearchFir(b) will remain
WOCONSTANIC, and when the brane is re-coordinated elsewhere, SearchFir(a) might resolve, which
would then allow SearchFir(b) to dereference further.

### Re-Coordination Example

```foolish
{
    fn = {a = x; b = a;};  // fn is WOCONSTANIC (a is CONSTANIC, b is WOCONSTANIC)

    x = 42;
    result = fn;           // Re-coordinate fn into new context
}
```

When `fn` is constanic-cloned for the assignment `result = fn`:

1. SearchFir(a) is cloned → transitions back to EMBRYONIC → searches for 'x' → finds 42 → becomes
   CONSTANT
2. SearchFir(b) is cloned → transitions to... what state?

**Ah, this is the subtlety!** When constanic-cloning SearchFir(b):
- SearchFir(b) is WOCONSTANIC
- It's waiting on SearchFir(a) which is CONSTANIC
- The clone should transition back to EMBRYONIC (re-search for 'a')
- OR the clone should check if its target (cloned SearchFir(a)) has resolved

**The Answer:** WOCONSTANIC clones revert to BRANING and receive messages from children.

### Constanic Cloning Protocol for Searches

When a WOCONSTANIC search is constanic-cloned into a new context:

1. **WOCONSTANIC search clone starts in BRANING state**
2. **Its children (targets) are constanic-cloned recursively:**
   - CONSTANT targets → referenced (not cloned)
   - CONSTANIC targets → cloned and transition to **EMBRYONIC** (re-dispatch searches)
   - WOCONSTANIC targets → cloned and transition to **BRANING** (receive messages from their children)

3. **The BRANING search receives messages** from its children:
   - When a CONSTANIC child transitions to EMBRYONIC, it re-searches in the new local context
   - The search may succeed in the new context (finds the missing binding)
   - The child sends a message back to the parent: "I resolved to CONSTANT"
   - The BRANING search receives this message and can now re-evaluate its dereferencing

4. **The search re-resolves** (calls `resolve()` again) when children change state

### Example: Constanic Cloning with Search Chain

```foolish
{
    fn = {a = x; b = a;};  // fn is WOCONSTANIC (a CONSTANIC, b WOCONSTANIC)

    x = 42;
    result = fn;           // Constanic-clone fn into new context
}
```

**Constanic cloning process:**

1. **Clone SearchFir(a)**:
   - Original is CONSTANIC (search for 'x' failed)
   - Clone transitions to **EMBRYONIC**
   - Clone re-searches for 'x' in new context → finds 42 → becomes CONSTANT

2. **Clone SearchFir(b)**:
   - Original is WOCONSTANIC (waiting on SearchFir(a))
   - Clone transitions to **BRANING**
   - Clone's target is cloned SearchFir(a) (now EMBRYONIC)
   - Clone waits for messages from cloned SearchFir(a)

3. **Cloned SearchFir(a) sends message** to cloned SearchFir(b):
   - "I resolved to CONSTANT (value = 42)"

4. **Cloned SearchFir(b) receives message**:
   - Re-calls `resolve()` to re-evaluate dereferencing
   - Searches for 'a' → finds cloned SearchFir(a)
   - Dereferences: SearchFir(a) is CONSTANT → dereference to value 42
   - SearchFir(b) becomes CONSTANT (value = 42)

**Result**: Both searches resolve successfully in the new context!

### Why BRANING (Not EMBRYONIC) for WOCONSTANIC Clones?

WOCONSTANIC searches **already found** what they were looking for. They don't need to re-search
from scratch. They just need to:
1. Wait for their children (dependencies) to re-resolve in the new context
2. Re-evaluate the dereferencing chain when dependencies change

BRANING is the perfect state for this:
- BRANING means "I'm done with my own work, waiting for children"
- BRANING receives messages from children (state change notifications)
- BRANING can transition to CONSTANIC when all children settle

### Constanic Cloning State Transitions Summary

| Original State | Clone State | Rationale |
|---------------|-------------|-----------|
| CONSTANT | Reference (no clone) | Immutable, safe to share |
| INDEPENDENT | Reference (no clone) | Immutable, safe to share |
| CONSTANIC | **EMBRYONIC** | Re-search in new context (local first, then beyond boundary if needed) |
| WOCONSTANIC | **BRANING** | Wait for children to re-resolve, then re-evaluate dereferencing |

### The Dereferencing Re-Evaluation

When a WOCONSTANIC search clone (in BRANING state) receives a message that a child changed state:

```java
void onChildStateChange(SearchFir child) {
    // Child changed state - re-evaluate dereferencing
    this.resolve(parent);  // Call resolve() again

    // resolve() will:
    // 1. Re-search for the identifier (finds same target, but target may have new state)
    // 2. Re-run dereferencing loop
    // 3. Update this search's state based on new dereferencing result
}
```

This works because `resolve()` is **idempotent** - it can be called multiple times and will
always produce the correct result based on current state of dependencies.

### CONSTANIC Re-Searches Locally First, Then Beyond Boundary

When a CONSTANIC search clone transitions to EMBRYONIC, it follows the standard search protocol:

1. **Search within the new brane's local scope** (brane-bounded backward search)
2. **If not found locally**, dispatch FulfillSearch message to parent (beyond boundary)
3. **Receive RespondToSearch** message from parent
4. **Transition to terminal state** based on result

This ensures that constanic clones get a fresh chance to find bindings in the new context.

### Current UBC2 Says

From CONSTANIC/WOCONSTANIC distinction:

> - **WOCONSTANIC**: Every search was found, but one or more of the found results are themselves

### What's Missing

The design doesn't specify **how** we determine "the found result is CONSTANIC." When a search

#### Proposal: Search Result State Propagation

When SearchFir resolves an identifier:

1. **Retrieve target FIR** from parent brane's statement array

   - `target.at_woconstanic()` → SearchFir becomes **WOCONSTANIC** (propagate waiting)

3. **Store reference** to target FIR (not a copy)

```java
public FIR getSearchResultFir() {
```

```java
public Object value() {
```

**Recommendation**: **Interpretation B** is more precise and matches UBC2's definitions:
- CONSTANIC: "nothing was found" (the identifier doesn't exist)

Add to UBC2 design, in EMBRYONIC section:

#### **Search Result Dereferencing**

When a search operation finds an identifier, it retrieves the target FIR from the parent brane's
statement array. The search's terminal state depends on the target's state:

| Target State | Search Becomes | Rationale |
|-------------|---------------|-----------|
| INDEPENDENT | CONSTANT | Immutable literal; safe to reference directly |
| CONSTANT | CONSTANT | Fully resolved; safe to reference directly |
| CONSTANIC | WOCONSTANIC | Identifier exists, but its value is unresolved (search failed for it) |
| WOCONSTANIC | WOCONSTANIC | Identifier exists, but it's waiting on dependencies |
| nigh (any pre-constanic) | Wait-for queue | Target still stepping; wait for it to reach constanic state |

```java
public FIR value() {

    }

}
```

**Special case**: When a search finds an identifier but the target is CONSTANIC, the search does
**not** become CONSTANIC (which would mean "identifier not found"). Instead, it becomes
WOCONSTANIC (which means "identifier found, but its value is unresolved"). This distinction
preserves semantic precision: CONSTANIC means the search itself failed; WOCONSTANIC means the
search succeeded but the result is not yet available.

## Issue 2 Analysis: WOCONSTANIC Race Condition

### The Scenario

```foolish
{
    a = x1 + x2 + x3 + x4 + x5;
    b = x1 + y1 + y2 + y3;
    c = a + b;
}
```

### Timeline

**PREMBRYONIC:**
- Statement 0: `a = BinaryOpChain(x1, x2, x3, x4, x5)` (system operator 🧠+)
- Statement 1: `b = BinaryOpChain(x1, y1, y2, y3)` (system operator 🧠+)
- Statement 2: `c = BinaryOp(a, b)` (system operator 🧠+)

Each BinaryOp/BinaryOpChain is a proto-brane containing SearchFirs for its operands.

**EMBRYONIC Step 1:**
- Statement 2 searches for `a` → **found** at statement 0
- Statement 2 searches for `b` → **found** at statement 1
- Statement 2's RHS: BinaryOp(🧠+) has both operands found
- Target `a` is **nigh** (still searching for x1, x2, x3, x4, x5)
- Target `b` is **nigh** (still searching for x1, y1, y2, y3)
- BinaryOp enters **wait-for queue** (waiting for a and b to become constanic)

**EMBRYONIC Step 2-10:**
- Statement 0 searches for x1, x2, x3, x4, x5 (search bandwidth limits)
- Statement 1 searches for x1, y1, y2, y3
- All searches dispatched to parent

**EMBRYONIC Step 11:**
- Parent responds: x1, x2, x3, x4, x5 not found
- Statement 0's BinaryOpChain: all operands CONSTANIC → transitions to **CONSTANIC**

**EMBRYONIC Step 12:**
- Parent responds: x1, y1, y2, y3 not found
- Statement 1's BinaryOpChain: all operands CONSTANIC → transitions to **CONSTANIC**

**EMBRYONIC Step 13:**
- Wait-for queue checks statement 2's BinaryOp
- Target `a` is now **CONSTANIC**
- Target `b` is now **CONSTANIC**
- BinaryOp(🧠+) has found both operands, but both are CONSTANIC
- BinaryOp transitions to **WOCONSTANIC**

### But What If c Checks Earlier?

**Alternative Timeline:**

**EMBRYONIC Step 5:**
- Wait-for queue checks statement 2's BinaryOp
- Target `a` is **EMBRYONIC** (still searching)
- Target `b` is **EMBRYONIC** (still searching)
- BinaryOp remains in wait-for queue

**EMBRYONIC Step 6:**
- Statement 0 finishes searching, all x's not found
- Statement 0 transitions to **CONSTANIC**
- Statement 1 still searching (only processed 2 of 4 identifiers due to search bandwidth)

**EMBRYONIC Step 7:**
- Wait-for queue checks statement 2's BinaryOp
- Target `a` is **CONSTANIC**
- Target `b` is **EMBRYONIC** (still searching)
- BinaryOp remains in wait-for queue

**EMBRYONIC Step 8:**
- Statement 1 finishes searching, all y's not found
- Statement 1 transitions to **CONSTANIC**

**EMBRYONIC Step 9:**
- Wait-for queue checks statement 2's BinaryOp
- Target `a` is **CONSTANIC**
- Target `b` is **CONSTANIC**
- BinaryOp transitions to **WOCONSTANIC**

### The Race

The key observation: **Statement c can observe a and b in different states depending on when the
wait-for check happens.**

Possibilities:
1. Both a and b are nigh → c remains in wait-for queue
2. a is CONSTANIC, b is nigh → c remains in wait-for queue
3. a is nigh, b is CONSTANIC → c remains in wait-for queue
4. Both a and b are CONSTANIC → c transitions to WOCONSTANIC

**This is actually fine!** The wait-for mechanism handles this correctly:
- While either dependency is nigh, keep waiting
- When both reach constanic states, check what those states are
- If both are CONSTANIC (or WOCONSTANIC), transition to WOCONSTANIC

### Is This Addressed in UBC2?

**Partially.** From EMBRYONIC Wait-For Mechanism:

> When a search result resolves to a non-constanic expression (the target is still being
> evaluated), the UBC2 implements a **wait-for mechanism**: the statement that depends on that
> expression is enqueued to wait for its dependencies to become constanic.

But it doesn't explicitly say:

1. **How often** wait-for checks happen (every EMBRYONIC step? every N steps?)
2. **What happens** when dependencies reach CONSTANIC vs WOCONSTANIC
3. **Whether** the expression can transition to WOCONSTANIC before all dependencies are CONSTANIC

### Design Clarification Needed

Add to UBC2 design, in EMBRYONIC Wait-For Mechanism section:

#### **Wait-For Queue Processing**

The wait-for queue is checked **every EMBRYONIC step** (subject to communication bandwidth limits).
For each expression in the wait-for queue:

1. **Check all dependencies**: Retrieve the current state of each dependency FIR
2. **If any dependency is nigh**: Keep the expression in the wait-for queue
3. **If all dependencies are constanic**:
   - If all dependencies are CONSTANT or INDEPENDENT → expression becomes **CONSTANT**
   - If any dependency is CONSTANIC or WOCONSTANIC → expression becomes **WOCONSTANIC**
   - Apply constanic cloning if needed (see Constanic Cloning section)

**Race condition handling**: An expression may observe its dependencies in various stages of
evaluation (nigh, EMBRYONIC, BRANING, constanic). This is expected. The wait-for mechanism ensures
that the expression only finalizes its state after all dependencies reach constanic states.

**Example**:
```foolish
{
    a = x1 + x2 + x3 + x4 + x5;  // Takes multiple steps to resolve
    b = x1 + y1 + y2 + y3;       // Takes multiple steps to resolve
    c = a + b;                    // Waits for both a and b
}
```

Timeline:
- c enters wait-for queue (a and b are nigh)
- Step N: a becomes CONSTANIC, b still EMBRYONIC → c remains in queue
- Step N+1: b becomes CONSTANIC → c becomes WOCONSTANIC

The race is resolved by the wait-for mechanism: c only transitions when **both** dependencies
reach constanic states.

## Issue 3: Dereferencing Chains During Sequencing

When the sequencer renders output, it must dereference search chains:

```foolish
{
    a = x;
    f = e;  // where e = d = c = b = a
}
```

Expected output:
```
{
＿a = ?;
＿b = ??;
＿c = ??;
＿d = ??;
＿e = ??;
＿f = ??;
}
```

But how does the sequencer know to render `??` vs `?`?

### Proposal: Recursive Dereferencing in Sequencer

```java
public String render(FIR fir) {
    if (fir.at_constanic()) {
        // Direct CONSTANIC: search failed
        return "?";
    }
    if (fir.at_woconstanic()) {
        // WOCONSTANIC: found something, but it's constanic
        // Could dereference to show what we're waiting on, or just render ??
        return "??";
    }
    if (fir.at_constant()) {
        return fir.value().toString();
    }
    if (fir.at_independent()) {
        return fir.value().toString();
    }
    throw new IllegalStateException("Cannot render nigh FIR");
}
```

**Alternative**: Dereference WOCONSTANIC to show the chain:

```java
public String render(FIR fir) {
    if (fir.at_woconstanic() && fir instanceof SearchFir) {
        FIR target = fir.getSearchResultFir();
        if (target.at_constanic()) {
            // Waiting on a CONSTANIC target: render as ? (propagate)
            return "?";
        }
        // Waiting on another WOCONSTANIC: render as ??
        return "??";
    }
    // ... rest
}
```

**Decision needed**: Should WOCONSTANIC searches that point to CONSTANIC targets render as `?` or
`??`?

**Recommendation**: Render as `??` to distinguish:
- `?` = "I searched and found nothing" (CONSTANIC)
- `??` = "I searched and found something, but it's not resolved" (WOCONSTANIC)

Even if the ultimate root is `?`, intermediate links should show `??` to indicate they
successfully found their target.

## Open Questions

### Q1: Should CONSTANIC Propagate Through Searches?

When `b = a` and `a = x` (x not found):

**Option A**: `b` becomes CONSTANIC (propagate "unfound" state)
- Rationale: The ultimate value is unfound
- Rendering: `a = ?; b = ?;`

**Option B**: `b` becomes WOCONSTANIC (found identifier, value is constanic)
- Rationale: Distinguish "couldn't find 'b'" from "found 'b', it's waiting on 'a'"
- Rendering: `a = ?; b = ??;`

**Recommendation**: **Option B** for semantic precision.

### Q2: How Deep to Dereference in Sequencer?

When rendering `f` which points to `e` which points to `d` ... which points to `a` which is `?`:

**Option A**: Always render immediate state (`f` is WOCONSTANIC → `??`)
**Option B**: Dereference to root and render root state (`f` → `e` → ... → `a` is CONSTANIC → `?`)

**Recommendation**: **Option A** for simplicity. The rendered state should reflect the FIR's own
state, not the transitive closure.

### Q3: Performance of Wait-For Checks

Checking the wait-for queue every EMBRYONIC step could be expensive if the queue is large.

**Optimization ideas**:
- Only check wait-for queue when a dependency state changes (callback-based)
- Batch wait-for checks (check every N steps, or when communication bandwidth is idle)
- Priority queue ordered by "closest to ready" (heuristic: count of nigh dependencies)

**Recommendation**: Start with simple "check every step" for correctness. Optimize later if needed.
